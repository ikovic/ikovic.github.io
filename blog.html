Async JavaScript - common problems and solutions

August 2, 2018

A few days ago I've stumbled across a very interesting article by Demir Selmanovic published on Toptal blog. The author goes over a brief history of asynchronous JavaScript, and compares the three existing approaches: callbacks, promises and async/await. That inspired me to think about my experience with async JavaScript because in my work I've had my share of all three approaches. Here are a few thoughts of my own on the same subject.

Generators - the missing link?

The first thing that came to mind was that the ES6 generators were omitted from the article. I understand the reasoning - it seems they never got enough popularity on their own. Based on the various reactions on social networks, I'd say that the generators by themselves require a certain way of thinking which is probably not natural for a web programmer dealing with business applications. Also, they do require some setup to solve the usual async problems. I'd still mention them, mainly because of these two reasons:

  • async/await is based on combining promises and generators
  • generators are an integral part of Redux Saga, a relevant package in Redux world
In my opinion, async/await solves the async problem good enough without any setup, so I don't see myself using generators directly if I have async/await available. Maybe they just got available in the wrong time - async/await came too fast and solved the same problem in a way that the majority of developers preferred.

Cancellation as the common pain point

There is a problem shared by all three approaches - it is not possible to cancel an async function once it was called. At least not in a standard way. How does that hurt us? Let's say you are implementing an autocomplete control, and you dispatch multiple requests as the user types in new characters. What if the response for the previous request comes after the response for the last request? If you did nothing to prevent that, you will end up with wrong search results displayed. So how do we fix this? Three approaches come to mind:

  • Use a library. There isn't a standard solution yet since there are trade offs in different ways to solve this problem. Bluebird is a powerful Promise polyfill which adds a lot of utility functions to Promise API, while CAF is a simpler solution, focused only on cancellation of async flows.
  • Roll your own solution. Even popular frameworks like next.js handle this problem in their own way.
  • Take a completely different approach and use streams. Angular uses observables by default, and they offer a built in way to cancel async operations. Just expect to read a lot and spend some time wrapping your mind around this.

Some of the presented approaches are just workarounds, so it's up to you to find an ideal solution for your specific problem. I hope this article can at least point you in the right direction :)

Handling Async Actions in Redux

September 10, 2018

This post is the first one in what should be a series of articles about the topics I'm currently interested at. Lately I've been trying to improve my architecture design skills, so the topics I'm learning about are how to structure and manage state using Redux, how to model and formalize internal state in React components, and how to properly test all of the above. I think that these are some of the key issues which can make or break a project. So let's start with a messy one: how to deal with async actions in Redux.

Async actions are not really supported out of the box with redux. You either have to roll with your own solution, or use a library like redux-thunk. Thunks are a part of official documentation, so they are probably the most approachable solution. In my opinion, the first problem that thunks don't solve is the cancellation of async requests. I've found two alternatives that help with that: redux-saga and redux-observable. Both come with trade offs: sagas are based on generators which I personally don't prefer, and redux-observable uses RxJs operators, which take some bundle size tax.

Regardless of library used, there is some thinking involved in designing the redux modules based around async actions. This article shows a nice way to get rid of the involved boilerplate.

Update: it seems I was wrong when I said that cancellation can't be done with thunks. There is a way, and it involves using redux middleware. The article above made me think about that. Middleware is a great architectural option when you want to have control over all of the dispatched actions. I see a few situations when you'd like to have that control to cancel some requests: if a certain kind of request is still waiting for a response, but a new request of that kind is issued, we can cancel the existing one by just removing it from the list of pending requests. Browser will still execute the request, but no callback will be fired. Most importantly, no unexpected change in state will happen. The other situation that comes to mind is cancelling all requests when navigating. That is also simple: clear all the pending requests or provide a black list argument of request types to cancel.

Formalizing component state

September 20, 2018

Using statecharts: TODO add links to project and references (davidkpiano talk, xstate, reference website, my github project). Experience with statecharts so far: hiearchical approach allows for modular approach. Complex states can have child state which simplifies things, creates black boxes where you need them. Main concept to swallow to be able to use this effectively is that when a transition is made on a parent state, child states will be first to receive the transition. In case they can handle that kind of transition, they will do it. Instead, the transition will bubble up to the first parent which can do that transition. Kind of similar to JS prototype inheritance. Nice things which xstate provides: guards (conditional transitions), external state...

Compared to reducers, state machines are pure functions which take the current state, and the wanted transition. New state is returned. That part is completely the same as with the reducers. But, state machines also define which transition is possible based on state and action, while in reducers you are much more free to do what you want. So state machines are more strict, which makes them tougher to develop and tougher to break unexpectedly.
